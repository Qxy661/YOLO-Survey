# 消融实验设计

> 消融实验（Ablation Study）是深度学习研究中验证各组件贡献的核心方法论。通过系统性地移除或替换模型中的某个模块，观察性能变化，从而定量评估每个设计决策的实际价值。没有消融实验的改进，只是"堆模块"；有消融实验的改进，才是"做研究"。

## 1. 什么是消融实验

消融实验源自神经科学术语"ablation"（切除），在深度学习中指：**系统性地移除或修改模型的某个组件，观察其对最终性能的影响**。

核心目的：

| 目的 | 说明 |
|:---|:---|
| **归因分析** | 明确性能提升来自哪个模块 |
| **避免虚假改进** | 排除"换了个随机种子就涨点"的偶然因素 |
| **指导设计** | 了解哪些组件是关键的，哪些可以去掉 |
| **论文说服力** | 审稿人最常问的问题之一就是"有没有消融实验" |

## 2. 消融实验设计原则

### 2.1 单变量原则

每次实验**只改变一个因素**，其他所有设置保持完全一致：

- 相同的数据集和划分
- 相同的训练超参数（epochs, lr, batch_size）
- 相同的硬件环境
- 相同的随机种子（或多次运行取平均）

> **反面案例**：同时换了 Backbone 和注意力模块，mAP 提升了 3%，无法判断是谁的贡献。

### 2.2 基线对照

必须有一个明确的 Baseline，所有改进都相对于它来衡量：

```
实验组结构：
├── Baseline（原始模型，无任何改动）
├── + 改动A（仅添加模块A）
├── + 改动B（仅添加模块B）
├── + 改动A + 改动B（同时添加）
```

### 2.3 多次运行取平均

深度学习训练存在随机性（数据增强、Dropout、权重初始化），单次结果不可靠：

| 指标 | 建议 |
|:---|:---|
| 运行次数 | 至少 3 次，推荐 5 次 |
| 报告方式 | 均值 +/- 标准差 |
| 统计检验 | 配对 t 检验或 Wilcoxon 检验 |

### 3.1 Backbone 消融

比较不同特征提取网络的效果：

| Backbone | 参数量 (M) | FLOPs (G) | mAP@0.5 | mAP@0.5:0.95 | 推理速度 (ms) |
|:---|:---:|:---:|:---:|:---:|:---:|
| CSPDarknet-S | 7.2 | 16.5 | 37.2 | 22.1 | 5.8 |
| EfficientNet-B0 | 5.3 | 12.8 | 36.8 | 21.7 | 6.2 |

| Swin-T | 28.3 | 45.6 | 39.5 | 24.2 | 12.1 |

**关注点**：不只是精度，还要比较参数量、计算量和推理速度的权衡。

|:---|:---:|:---:|:---:|
| Baseline | 37.2 | -- | 5.8ms |

| + ECA | 37.5 | +0.3 | 5.9ms |

**特征融合模块消融示例：**

| Neck 配置 | mAP@0.5 | 参数量 |
|:---|:---:|:---:|

## 4. 结果分析方法

### 4.1 归因分析

对每个消融实验的结果进行归因：

```
总提升 = Baseline 改进量

各模块贡献：
├── 模块A 贡献 = (Baseline + A) - Baseline
├── 模块B 贡献 = (Baseline + B) - Baseline
├── 交互效应 = (Baseline + A + B) - Baseline - A贡献 - B贡献
```

**示例归因分析：**

| 配置 | mAP@0.5 | 增量 |
|:---|:---:|:---:|
| Baseline | 37.2 | -- |
| + CBAM | 37.4 | +0.2 |
| + BiFPN | 37.8 | +0.6 |
| + CBAM + BiFPN | 37.9 | +0.7（交互效应 = -0.1）|

交互效应为负，说明 CBAM 和 BiFPN 存在一定冲突。

### 4.2 增量贡献分析

计算每个组件的边际贡献：

| 组件 | 独立贡献 | 边际贡献（在最优组合上） |
|:---|:---:|:---:|
| 新 Backbone | +1.2 | +1.2 |
| 注意力模块 | +0.5 | +0.3 |
| 新损失函数 | +0.7 | +0.5 |
| 数据增强 | +2.1 | +1.8 |

## 6. 注意事项

### 6.1 控制变量

- 训练曲线也应对比，不只是最终指标
- 确保数据预处理完全一致（归一化、resize 方式）
- 注意 PyTorch 版本、CUDA 版本的一致性
- 使用 `torch.backends.cudnn.deterministic = True` 保证可复现性

### 6.2 计算开销评估

除了精度指标，必须报告效率指标：

| 指标 | 测量方法 | 意义 |
|:---|:---|:---|
| 参数量 | `model.parameters()` | 模型大小 |
| FLOPs | `thop` 工具 | 计算量 |
| 推理速度 | 预热 + 多次平均 | 实际部署性能 |
| 显存占用 | `torch.cuda.max_memory_allocated()` | 硬件需求 |
| 训练时间 | 训练总时长 | 研发成本 |

### 6.3 统计显著性

单次实验结果不可靠，需要统计检验：

```python
# 配对 t 检验示例
from scipy import stats

baseline_runs = [0.372, 0.370, 0.374, 0.371, 0.373]
improved_runs = [0.378, 0.376, 0.380, 0.377, 0.379]

t_stat, p_value = stats.ttest_rel(improved_runs, baseline_runs)
print(f"p-value: {p_value:.4f}")
# p < 0.05 说明差异具有统计显著性
```

### 6.4 常见陷阱

| 陷阱 | 说明 | 解决方案 |
|:---|:---|:---|
| Cherry-picking | 只报告最好的结果 | 多次运行取平均 |
| 选择性报告 | 隐藏不好的消融结果 | 报告所有消融实验 |
| 超参数不公平 | 改进模型调了超参，baseline 没调 | 统一调参策略 |
| 数据泄露 | 测试集信息泄露到训练中 | 严格划分数据集 |
| 忽略效率 | 只看精度不看速度 | 同时报告效率指标 |
