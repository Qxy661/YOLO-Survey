# 高分辨率策略：让小目标"无处藏身"

> **阅读时间**: ~10 min | **前置知识**: 多尺度特征融合（FPN / PANet）

在无人机航拍、遥感影像、远距离监控等场景中，目标在图像中仅占几十甚至几个像素。
当输入分辨率不足以覆盖这些微小目标时，再精巧的网络结构也难以奏效——信息在采样阶段就已丢失。
本专题系统梳理**高分辨率策略**，从最直接的提升输入分辨率，到 SAHI 切片推理、超分辨率预处理、多尺度训练/测试，以及 P2 高分辨率检测头等进阶方案。

---

## 1. 输入分辨率的影响

### 1.1 从 320 到 1280：分辨率如何影响小目标检测

YOLO 系列的默认输入分辨率通常为 640×640。对于 COCO 中 "small" 类别（面积 < 32×32 像素），目标在输入图像上可能只有 10-20 像素；但在 VisDrone、DOTA 等数据集中，目标可能仅有 4-8 像素。

提升输入分辨率（`imgsz`）是最直观的策略：

| imgsz | 特征图最小步长对应分辨率 | 小目标可感知像素 | 参数量变化 | 典型 mAP@0.5 (VisDrone) | 典型 FPS (RTX 3090) |
|:-----:|:----------------------:|:---------------:|:---------:|:----------------------:|:-------------------:|
| 320   | 320/8 = 40×40 (P3)    | 2-4 px          | 基准       | ~18.2                  | ~180                |
| 640   | 640/8 = 80×80 (P3)    | 4-8 px          | 基准       | ~28.5                  | ~120                |
| 1280  | 1280/8 = 160×160 (P3) | 8-16 px         | 基准       | ~35.8                  | ~45                 |
| 1920  | 1920/8 = 240×240 (P3) | 12-24 px        | 基准       | ~38.1                  | ~20                 |

> 注：以上数据为典型参考值，实际数值因模型变体、数据集和硬件环境而异。

### 1.2 计算量与精度的 Trade-off

提升分辨率的代价是**二次方级计算增长**：

- **FLOPs** 与输入面积成正比：`imgsz` 从 640 翻倍到 1280，FLOPs 大约增加 4 倍。
- **显存占用** 同样呈二次增长，尤其在 Backbone 的浅层特征图上。
- **推理延迟** 线性增长（受内存带宽限制时甚至超线性）。

```
FLOPs(imgsz) ≈ FLOPs(640) × (imgsz / 640)^2
```

因此，"暴力提升分辨率" 并非万能解。需要配合以下策略来平衡精度与效率。

## 2. SAHI：切片辅助超推理

### 2.1 核心原理

**SAHI (Slicing Aided Hyper Inference)** 的思路非常简单：

1. 将高分辨率大图按固定大小切成若干**重叠切片 (slices)**。
2. 对每个切片分别执行常规目标检测推理。
3. 将所有切片的检测结果映射回原图坐标。
4. 使用 **NMS (Non-Maximum Suppression)** 合并重叠区域的冗余检测框。

```
┌─────────────────────────┐
│  原图 (4000×3000)       │
│  ┌──────┐              │
│  │slice1│──┐           │
│  └──────┘  │           │
│       ┌────▼──┐        │
│       │slice2 │        │
│       └───────┘        │
│  每个 slice 独立推理    │
│  最后合并所有结果        │
└─────────────────────────┘
```

### 2.2 关键参数

| 参数 | 含义 | 推荐值 | 说明 |
|:----:|:----:|:-----:|:----:|
| `slice_size` | 切片边长（像素） | 512-640 | 通常设为模型训练时的 imgsz |
| `overlap_ratio` | 相邻切片的重叠比例 | 0.2-0.4 | 重叠越大，边界处漏检越少，但推理量增加 |
| `nms_threshold` | NMS 的 IoU 阈值 | 0.5 | 用于合并跨切片的重复检测 |
| `postprocess_type` | 后处理方式 | `"NMS"` 或 `"NMM"` | NMM (Non-Maximum Merging) 可软合并置信度 |

### 2.3 SAHI 的核心优势

- **即插即用**：无需重新训练模型，任何预训练检测器均可直接使用。
- **对小目标效果显著**：本质上等价于用小 imgsz 模型在大图上密集滑窗检测。
- **显存友好**：每次只处理一个 slice，峰值显存与单张小图推理相同。
- **可与任何检测器组合**：YOLO、RT-DETR、DETA、Florence-2 等均可配合使用。

### 2.4 实际效果

在 **VisDrone** 数据集上的典型对比：

| 方法 | imgsz | mAP@0.5 | mAP@0.5:0.95 | FPS |
|:----:|:-----:|:-------:|:------------:|:---:|
| YOLOv8n baseline | 640 | 23.1 | 12.8 | 145 |
| YOLOv8n + SAHI (slice=640, overlap=0.3) | 原图 | 34.7 | 19.5 | 38 |
| YOLOv8n imgsz=1280 | 1280 | 30.2 | 16.9 | 52 |
| YOLOv8n imgsz=1280 + SAHI | 原图 | 38.5 | 22.1 | 15 |

> SAHI 在不改变模型的情况下，mAP@0.5 提升约 **11 个百分点**，且无需任何额外训练。

### 2.5 代码示例

```python
from sahi import AutoDetectionModel
from sahi.predict import get_sliced_prediction

# 加载模型
detection_model = AutoDetectionModel.from_pretrained(
    model_type="ultralytics",
    model_path="yolov8n.pt",
    confidence_threshold=0.4,
    device="cuda:0",
)

# 切片推理
result = get_sliced_prediction(
    image="high_res_aerial.jpg",
    detection_model=detection_model,
    slice_height=640,
    slice_width=640,
    overlap_height_ratio=0.3,
    overlap_width_ratio=0.3,
)

# 可视化结果
result.export_visuals(export_dir="output/")
```

## 3. 超分辨率预处理

### 3.1 思路：先放大，再检测

当原始图像本身分辨率不足时（如低分辨率监控画面），可以先用 **超分辨率 (Super-Resolution, SR)** 模型将图像放大，再送入检测器。

```
低分辨率图像 → SR 模型 (×4 放大) → 高分辨率图像 → 检测器 → 检测结果
```

### 3.2 常用 SR 模型

| 模型 | 放大倍数 | 特点 | 推理速度 (512×512→2048×2048) |
|:----:|:-------:|:----:|:--------------------------:|
| Real-ESRGAN | ×2/×4 | GAN-based，细节恢复强，通用性好 | ~120ms (GPU) |
| SwinIR | ×2/×4 | Transformer-based，PSNR 高 | ~200ms (GPU) |
| ESRGAN | ×4 | 经典 GAN 方案 | ~80ms (GPU) |
| StableSR | ×4 | 基于 Stable Diffusion，感知质量极高 | ~800ms (GPU) |
| FastSRN | ×4 | 轻量级，移动端友好 | ~30ms (GPU) |

### 3.3 计算开销分析

SR 预处理的计算开销**不可忽视**：

| 流程 | 分辨率变化 | 耗时 (RTX 3090) |
|:----:|:---------:|:---------------:|
| 原图直接检测 | 640×640 | ~8ms |
| Real-ESRGAN ×4 + 检测 | 256×256 → 1024×1024 → 640×640 检测 | ~120ms + 8ms |
| 直接高分辨率检测 | 1280×1280 | ~22ms |

**结论**：SR 预处理通常比直接提升 imgsz 更慢，但在以下场景有独特价值：

1. **原始图像质量极差**（模糊、噪声、压缩伪影严重），提升 imgsz 无法恢复丢失信息。
2. **检测器无法更换或重新训练**，只能通过预处理改善输入质量。
3. **与 SAHI 结合**：先 SR 放大，再切片检测，可进一步提升极小目标的召回率。

### 3.4 SR + 检测的联合训练

近年研究（如 `Detection-SR`, `SRDet`）探索了将 SR 与检测**端到端联合训练**的方式：

- SR 模块不追求像素级 PSNR 最优，而是以检测 loss 为导向优化。
- 可避免 SR "过度美化" 导致的伪影干扰检测。
- 但训练复杂度显著增加，目前在工业界尚未普及。

## 4. 多尺度训练与测试

### 4.1 多尺度训练 (Multi-scale Training)

在训练过程中，**每个 batch 随机选择不同的输入分辨率**，增强模型对不同尺度目标的适应能力。

典型实现方式（以 YOLO 为例）：

```python
# 每隔 N 个 batch，随机调整 imgsz
# 基准: 640, 范围: [0.5, 1.5] × 640 = [320, 960]
# 实际 imgsz 取 32 的倍数
if i % 10 == 0:
    new_imgsz = random.choice([320, 384, 448, 512, 576, 640, 704, 768, 832, 896, 960])
    dataset.imgsz = new_imgsz
```

**效果**：

| 训练策略 | mAP@0.5 | mAP@0.5:0.95 | 小目标 AP |
|:-------:|:-------:|:------------:|:---------:|
| 固定 640 | 52.3 | 34.1 | 18.5 |
| 多尺度 [320-960] | 53.8 | 35.6 | 20.2 |
| 多尺度 [320-1280] | 54.1 | 36.0 | 21.3 |

### 4.2 多尺度测试时增强 (Multi-scale TTA)

在推理时，对同一张图像在**多个分辨率下分别检测**，再合并结果：

```
原图 → imgsz=640 检测 ─┐
原图 → imgsz=960 检测 ─┤→ 合并 (NMS) → 最终结果
原图 → imgsz=1280 检测 ─┘
```

常见 TTA 策略：

| TTA 组合 | 精度提升 | 推理时间倍数 |
|:--------:|:-------:|:----------:|
| 640 + 960 | +1.2 mAP | ×2 |
| 640 + 960 + 1280 | +2.0 mAP | ×3 |
| 640 + 960 + 1280 + 翻转 | +2.8 mAP | ×6 |

> TTA 的收益递减明显，通常**2-3 个尺度**即可获得大部分收益。在竞赛中常用，但在实时部署中因延迟过高而不实用。

### 4.3 Scale YOLOv5 / YOLOv9 中的实现

YOLOv5/YOLOv9 官方已内置多尺度训练：

```python
# train.py 中的 multi_scale 参数
parser.add_argument('--multi-scale', action='store_true',
                    help='adjust img_size every 10 batches (0.5x to 1.5x)')
```

启用后，训练日志会显示 `img_size` 在范围内周期性变化。

## 5. 高分辨率特征保持：P2 检测头

### 5.1 什么是 P2 检测头

标准 YOLO 检测头使用 P3、P4、P5 三个尺度的特征图：

| 检测头 | 特征图尺寸 (imgsz=640) | 步长 | 负责目标尺寸 |
|:-----:|:---------------------:|:---:|:----------:|
| P3    | 80×80                 | 8   | 小目标      |
| P4    | 40×40                 | 16  | 中目标      |
| P5    | 20×20                 | 32  | 大目标      |

如果增加 **P2 检测头**，则：

| 检测头 | 特征图尺寸 (imgsz=640) | 步长 | 负责目标尺寸 |
|:-----:|:---------------------:|:---:|:----------:|
| **P2** | **160×160** | **4** | **极小目标** |
| P3    | 80×80                 | 8   | 小目标      |
| P4    | 40×40                 | 16  | 中目标      |
| P5    | 20×20                 | 32  | 大目标      |

### 5.2 P2 在 640 输入下为何效果不明显

直觉上，160×160 的高分辨率特征图应该对小目标帮助很大。但实验表明，在 **640×640 输入下，P2 的收益有限甚至为负**：

| 配置 | mAP@0.5:0.95 | 小目标 AP | 推理速度 |
|:----:|:------------:|:---------:|:-------:|
| P3-P5 (baseline, imgsz=640) | 37.4 | 21.8 | 120 FPS |
| P2-P5 (imgsz=640) | 37.1 | 21.5 | 95 FPS |
| P3-P5 (imgsz=1280) | 41.2 | 27.6 | 48 FPS |
| P2-P5 (imgsz=1280) | 42.8 | 29.9 | 35 FPS |
| P2-P5 (imgsz=1920) | 44.1 | 32.5 | 18 FPS |

原因分析：

1. **信息瓶颈在输入端**：当 imgsz=640 时，一个 8px 的小目标在原图中可能对应真实世界的几十厘米到几米。目标本身的像素信息已经不足，P2 层的高分辨率特征图只是在"放大噪声"。
2. **感受野过小**：P2 的有效感受野很小，缺乏上下文信息，容易产生大量误检。
3. **计算开销增加**：P2 特征图面积是 P3 的 4 倍，增加 P2 头会使检测头参数量和计算量翻倍，却无法获得相应的精度提升。

### 5.3 P2 的正确打开方式：高分辨率输入

P2 检测头需要配合**足够大的输入分辨率**才能发挥优势。当 imgsz >= 1024 时：

- 原图中的小目标包含足够多的像素（16-32px），P2 层能提取到有意义的特征。
- P2 的高分辨率特征图能够精确定位这些小目标的边界。
- 与 P3-P5 形成互补的多尺度覆盖。

**推荐配置**：

| 场景 | 推荐方案 |
|:----:|:-------:|
| 实时检测 (640 输入) | P3-P5 即可，不必加 P2 |
| 离线高精度 (1280+ 输入) | P2-P5，小目标 AP 提升 2-5 个点 |
| 极高分辨率 (1920+ 输入) | P2-P5 + SAHI，最大化小目标召回 |

### 5.4 YOLOv8-P2 修改示例

```yaml
# yolov8-p2.yaml (修改检测头配置)
# 原始 P3-P5 头 → 改为 P2-P5 头

head:
  - [-1, 1, nn.Upsample, [None, 2, "nearest"]]    # 新增: 上采样到 P2 分辨率
  - [[-1, 2], 1, Concat, [1]]                      # 与 Backbone P2 层拼接
  - [-1, 3, C2f, [256]]                             # P2 特征处理

  - [-1, 1, nn.Upsample, [None, 2, "nearest"]]
  - [[-1, 5], 1, Concat, [1]]
  - [-1, 3, C2f, [256]]                             # P3

  - [-1, 1, nn.Upsample, [None, 2, "nearest"]]
  - [[-1, 8], 1, Concat, [1]]
  - [-1, 3, C2f, [512]]                             # P4

  - [-1, 1, Conv, [512, 3, 2]]
  - [[-1, 11], 1, Concat, [1]]
  - [-1, 3, C2f, [1024]]                            # P5

  - [[P2_out, P3_out, P4_out, P5_out], 1, Detect, [nc]]
```

## 6. 策略对比与选择指南

| 策略 | 是否需要训练 | 精度提升 | 计算开销 | 适用场景 |
|:----:|:----------:|:-------:|:-------:|:-------:|
| 提升 imgsz | 否（或微调） | 显著 | 高（二次增长） | 通用，显存足够时首选 |
| SAHI 切片推理 | 否 | 显著 | 中（取决于切片数） | 大图检测，即插即用 |
| 超分辨率预处理 | 否 | 中等 | 高 | 低分辨率/模糊图像 |
| 多尺度训练 | 是 | 中等 | 低（训练时间增加） | 通用，建议始终启用 |
| 多尺度 TTA | 否 | 轻微 | 高（倍数增长） | 竞赛、离线评估 |
| P2 检测头 | 是 | 中等 | 中 | 配合高分辨率输入使用 |

**实践建议**：

1. **第一步**：确保启用了多尺度训练（几乎零成本的收益）。
2. **第二步**：根据硬件能力适当提升 imgsz（640→1280 是性价比最高的跳跃）。
3. **第三步**：对于大图场景，叠加 SAHI 切片推理。
4. **第四步**：如果小目标仍不够理想，考虑添加 P2 检测头并配合 1024+ 输入。
5. **第五步**：仅在图像质量确实很差时，才考虑 SR 预处理。

## 思考题

<details>
<summary>1. 为什么 SAHI 的 overlap_ratio 不能设得太小？设为 0 会有什么问题？</summary>

overlap_ratio 设为 0 时，相邻切片无重叠区域。如果目标恰好落在切片边界上，会被切成两半，导致两个切片都无法完整检测到该目标（目标可见部分不足，置信度过低被过滤）。适当增加 overlap（0.2-0.4）可以确保边界处的目标在至少一个切片中被完整覆盖。

</details>

<details>
<summary>2. 对于 4K 监控视频中的行人检测，你会选择哪种策略组合？为什么？</summary>

推荐组合：**SAHI（slice_size=640, overlap=0.3）+ 轻量模型（如 YOLOv8n）**。理由：4K 图像（3840×2160）直接输入模型会导致显存溢出且推理极慢；SAHI 可以将问题分解为多个 640×640 的子问题，每个切片推理快速，且无需额外训练。如果还需要进一步提升，可在 SAHI 前加入 Real-ESRGAN 处理压缩伪影严重的区域。

</details>

<details>
<summary>3. P2 检测头在 imgsz=640 时效果不明显，是否意味着 P2 完全无用？</summary>

不完全是。P2 在 640 输入下效果不明显的原因是输入分辨率不够，而非 P2 结构本身有问题。当 imgsz 提升到 1280 或更高时，P2 可以带来 2-5 个点的小目标 AP 提升。此外，在某些特定场景（如文档中的微小文字检测）中，即使 640 输入下 P2 也可能有帮助——关键在于目标是否在 4× 步长的特征图上有足够的信息。

</details>

<details>
<summary>4. 超分辨率预处理和直接提升 imgsz，本质区别是什么？</summary>

直接提升 imgsz 是让检测器在更高分辨率下**重新提取特征**，图像本身的信息量不变（只是采样更密集）。超分辨率预处理则是通过学习到的先验**生成/补全高频细节**，在理想情况下可以恢复图像中因降采样而丢失的信息。因此，当原始图像质量很差时，SR 预处理可能比简单提升 imgsz 更有效——但 SR 也可能引入伪影，需要谨慎评估。

</details>

<details>
<summary>5. 如果要在嵌入式设备（如 Jetson Orin Nano）上部署小目标检测，你会如何选择高分辨率策略？</summary>

嵌入式设备算力有限，应优先考虑**计算效率最高的方案**：(1) 使用多尺度训练增强模型对小目标的鲁棒性（零推理成本）；(2) imgsz 设为 640 或 960（权衡精度与速度）；(3) 对于大图输入使用 SAHI（显存友好，可逐片推理）；(4) 避免使用 SR 预处理和 TTA（计算开销过大）；(5) 可考虑使用 TensorRT 优化模型推理速度。

</details>
